# Working with decisions

Decisions (DEC-NNNN) are the load-bearing artifact in Cairn. This
page covers everything about creating, accepting, querying, and
designing them. If you only read one guide besides
[concepts.md](concepts.md), make it this one.

The conceptual basics (what a DEC is, how scope works, the
supersedes chain) are in [`concepts.md`](concepts.md). Here we go
deeper: the three creation paths, the file format in detail, the
acceptance flow, querying patterns, and scope-design heuristics.

---

## Three ways decisions get created

### 1. Adoption ingestion

The most decisions you'll ever see at once. During adoption, three
phases propose DEC drafts:

- **Phase 8 (docs ingest).** Reads every `*.md` in `docs/` and
  `README.md`. For each section the Haiku classifier judges as
  "expressing an architectural choice", a draft is written.
- **Phase 9 (source comments).** Walks essay-class block comments in
  source files (>3 lines or >200 chars or JSDoc with >30 words of
  prose). For each block classified as "rationale", a draft is
  written.
- **Phase 10 (rules merge).** Walks `CLAUDE.md` / `AGENTS.md` /
  `.claude/rules/*.md`. For each H2 section that expresses a rule,
  a draft is written.

A typical first adoption produces 50-200 drafts. You triage them in
the post-adoption attention drain.

The draft body is generated by Haiku; the title and rationale are
the model's best attempt at extracting the structure. You can edit
freely before accepting.

### 2. Reviewer subagent

After every multi-chunk task, the `cairn:reviewer` subagent reads
the diff and looks for non-obvious choices the implementation made.
Examples of what counts as "non-obvious":

- A picked-out specific value (rate limit = 10/min, retry count = 5,
  backoff base = 1s).
- A choice between equally-reasonable alternatives the operator
  didn't specify (`argon2id` vs `bcrypt`, `crypto.randomUUID()` vs
  `nanoid`).
- A pattern that's repeated across the diff (every new error path
  uses the new `domain-error` envelope, suggesting a captured
  convention).
- A constraint the implementation respected that wasn't in the spec
  (refunds reused the existing idempotency-key middleware).

For each, the reviewer drafts a DEC via `cairn_record_decision`. The
draft surfaces in the stop hook for review.

### 3. Manual via `cairn_record_decision`

When you know in advance that something needs recording, drop it
explicitly. From within Claude Code:

> Record a decision: title "Refund retries use exponential backoff
> with jitter, max 5 attempts, base 1s." Scope
> `packages/billing/refunds/**`. Rationale: "Aligns with the rest of
> the billing retry strategy, capped at 5 to avoid runaway hot loops."

Claude calls `cairn_record_decision` with those fields. The MCP
server allocates the next `DEC-NNNN`, writes the draft, and returns
the id and path. The draft surfaces in the stop hook the same way
reviewer-extracted drafts do.

You can also call from a CLI:

```bash
cairn mcp call cairn_record_decision '{
  "title": "Refund retries use exponential backoff with jitter, max 5 attempts, base 1s",
  "summary": "Aligns with the rest of the billing retry strategy.",
  "scope_globs": ["packages/billing/refunds/**"],
  "body_markdown": "## Rationale\n\nMatches the existing pattern in packages/billing/charges/. Cap of 5 prevents runaway loops if the upstream is unresponsive."
}'
```

When to do this: **before** the agent runs the implementation. The
goal is recording the choice while it's fresh in your head, so the
agent's implementation is constrained by the recorded rule rather
than the agent inventing its own and the reviewer extracting it
post-hoc.

## What a decision file looks like

Every accepted DEC is a markdown file with YAML frontmatter:

```yaml
---
id: DEC-0042
title: Auth tokens expire after 24 hours
type: adr
status: accepted
audience: dual
generated: 2026-04-12T18:23:00Z
verified-at: 2026-05-01T18:23:00Z
decided_at: 2026-04-12
decided_by: operator
scope_globs:
  - src/auth/**
  - packages/api/src/middleware/auth/**
supersedes: DEC-0017
superseded_by: null
assertions:
  - id: a1
    kind: text_must_match
    pattern: "expiresIn:\\s*'24h'"
    in_globs: ["src/auth/jwt.ts"]
human_review_hint: |
  When changing the lifetime, update the silent-refresh middleware
  consumers and notify mobile clients via the changelog.
related_invariants: [INV-0042]
---

# DEC-0042 — Auth tokens expire after 24 hours

## Context

PCI compliance audit (Q1 2026) flagged our prior 7-day refresh
window as out of policy for cardholder-data scope. Short-lived
bearer tokens with refresh-on-active-session is the agreed
remediation.

## Decision

All bearer tokens issued by `src/auth/` expire 24 hours after issue.
Refresh tokens follow the same lifetime. Sessions that need longer
lifetime use the silent-refresh middleware in
`packages/api/src/middleware/auth/silent-refresh.ts`.

## Consequences

- Mobile clients: must implement silent-refresh or re-prompt for
  login daily.
- Service-to-service: use the `service-account` flow which has its
  own 90-day rotation policy (covered by DEC-0019).
- Old test fixtures with 7-day token lifetimes need updating —
  baseline scan flagged 14 files.
```

### Frontmatter fields

| Field                | Type            | Required | Notes                                                                       |
| -------------------- | --------------- | -------- | --------------------------------------------------------------------------- |
| `id`                 | string          | yes      | `DEC-NNNN` zero-padded, server-allocated.                                   |
| `title`              | string          | yes      | One-line, ≤80 chars. Rendered everywhere — keep specific.                   |
| `type`               | string          | yes      | Always `adr` for decisions.                                                 |
| `status`             | string          | yes      | `draft`, `accepted`, `superseded`, `archived`.                              |
| `audience`           | string          | yes      | `dual` (default), `ai-only`, or `human-only`.                               |
| `generated`          | ISO 8601        | yes      | When the file was first written.                                            |
| `verified-at`        | ISO 8601        | yes      | Last time GC confirmed the rationale still matches the codebase.            |
| `decided_at`         | ISO 8601 date   | yes      | When the decision was made (may differ from `generated`).                   |
| `decided_by`         | string          | yes      | Usually `operator`; can be a contributor handle.                            |
| `scope_globs`        | string[]        | yes      | The file globs this decision applies to. See "scope design" below.          |
| `supersedes`         | string \| null  | no       | DEC id this replaces. Must exist in ledger or be a legacy `<id>-<suffix>`.  |
| `superseded_by`      | string \| null  | auto     | Set by Cairn when a newer DEC supersedes this one.                          |
| `assertions`         | object[]        | no       | Machine-checkable rules — see "assertions" below.                           |
| `human_review_hint`  | string          | no       | Note for future humans — what to consider when revising this decision.      |
| `related_invariants` | string[]        | no       | Linked §INV ids. Lens uses these for cross-references.                      |

### Body sections

The body is free markdown. Convention:

```
## Context        — what made this decision necessary
## Decision       — the actual choice, in 1-3 paragraphs
## Consequences   — downstream effects: who needs to do what
```

Other sections (Alternatives, Notes, References) are fine when the
decision warrants them. Keep it brief — the agent quotes from this
text on every related prompt; long bodies dilute the signal.

### Assertions

Assertions are machine-checkable rules attached to the decision.
The decision-assertions sensor (Layer C) evaluates them against
every diff that touches in-scope files. Available kinds:

| Kind                          | What it checks                                                                |
| ----------------------------- | ----------------------------------------------------------------------------- |
| `schema_must_contain`         | Specific column + type exists in a named DB table.                            |
| `text_must_match`             | A regex pattern appears at least once in files matching `in_globs`.           |
| `text_must_not_match`         | A regex pattern appears nowhere in files matching `in_globs`.                 |
| `index_must_exist`            | Named DB index exists with specific columns + optional WHERE clause.          |
| `query_must_filter_by`        | Every SELECT/UPDATE/DELETE on a table includes specific columns in WHERE.     |
| `route_must_have_guard`       | Every route in `in_globs` declares a named auth guard.                        |
| `event_must_emit`             | Method-X always emits event-Y with specified payload keys.                    |
| `service_method_must_call`    | Method-X always calls method-Y before returning.                              |

Most decisions don't need assertions. Use them when:

- You've been bitten by the rule being silently violated before.
- The rule is critical (security, data integrity, regulatory).
- The pattern is mechanical enough to express as one of the above.

Pure-rationale decisions (most of them) carry no assertions — they
work by being loaded into the agent's context, not by mechanical
enforcement.

## Accepting, rejecting, editing

Drafts live at `.cairn/ground/decisions/_inbox/<id>.draft.md`.
They're invisible to the daily flow until accepted.

### The attention prompt

When the stop hook surfaces a draft (or `cairn-attention` drains the
queue), you get an `AskUserQuestion` panel:

```
DEC-0099 — Refund rate limit: 10/min/user

Rationale: Implementation chose 10 requests/minute per user, which
is below the global API limit (60/min) and matches the pattern in
DEC-0067 for transactional endpoints.

[a] accept   [b] reject   [c] edit first
```

### `[a]` accept

The draft is moved from `_inbox/` to the canonical zone:

```
.cairn/ground/decisions/_inbox/DEC-0099.draft.md  →  
.cairn/ground/decisions/DEC-0099.md
```

`status: draft` becomes `status: accepted`. `decisions.ledger.yaml`
is rebuilt under the per-write `flock`. An invalidation event is
written to `.cairn/events/` so any other live sessions know the
ledger changed.

If the draft came from a source-comment ingest (Phase 9), accept
also strips the originating essay comment and replaces it with a
bare `§DEC-NNNN` symbol on the same line. (Phase 12 typically
handles this earlier, but if you accept a draft that wasn't part of
a strip-replace pass, accept does it now.)

### `[b]` reject

The draft is renamed:

```
.cairn/ground/decisions/_inbox/DEC-0099.draft.md  →  
.cairn/ground/decisions/_inbox/DEC-0099.rejected.md
```

The id stays reserved — Cairn never recycles ids. You're prompted
for a one-line reason that gets appended to the rejected file.

If you later realize the rejection was wrong, you can restore the
draft:

```
cairn_resolve_attention({
  kind: "decision_draft",
  item_id: "DEC-0099",
  choice: "a"   // accept
})
```

The MCP tool transparently locates the rejected file and acts on
it. Response includes `auto_restored_from: "rejected"`.

### `[c]` edit first

The skill renders the full draft body inline as a chat message:

````
**Editing DEC-0099.**

```markdown
---
id: DEC-0099
title: Refund rate limit: 10/min/user
status: draft
scope_globs:
  - packages/billing/refunds/**
---

# DEC-0099 — Refund rate limit: 10/min/user

## Rationale

Implementation chose 10 requests/minute per user...
```
````

Then a follow-up panel:

```
[a] Rewrite the title
[b] Rewrite the rationale (body)
[c] Rewrite both
[d] Cancel — keep as-is and re-prompt accept/reject
```

You pick which field. The skill asks you for the replacement text
inline:

```
> Replacement title for DEC-0099?
```

You type. The skill applies the edit via `Edit` against the
draft file, re-renders the new body, and re-prompts accept / reject /
edit again. You stay in the chat the whole time.

### When to edit

Most drafts the reviewer extracts have decent wording. Edit when:

- **Title is too generic.** "Use a rate limit" is uselessly broad;
  "Refund rate limit: 10/min/user" is specific.
- **Rationale is missing the why.** "Picked 10/min" doesn't help —
  you want "Picked 10/min because that's the average max usage in
  the existing analytics, +50% headroom."
- **Scope is wrong.** The reviewer guessed `packages/billing/**` but
  the rule should only apply to refunds. Fix the scope before
  accepting; once accepted, you'd have to write a superseder to
  narrow.

## The supersedes chain

You never edit an accepted DEC. When the project changes its mind,
you write a new DEC that supersedes the old one.

### Writing a superseder

```bash
cairn mcp call cairn_record_decision '{
  "title": "Auth tokens expire after 8 hours (PCI Level 1)",
  "summary": "Escalated from 24h after the Q2 audit reclassified us as Level 1.",
  "scope_globs": ["src/auth/**", "packages/api/src/middleware/auth/**"],
  "supersedes": "DEC-0042",
  "body_markdown": "## Context\n\nThe Q2 PCI audit reclassified us as Level 1 (>6M transactions/year), which mandates an 8-hour cap.\n\n## Decision\n\nAll bearer tokens expire 8 hours after issue. Refresh tokens follow the same lifetime."
}'
```

The new DEC's frontmatter has `supersedes: DEC-0042`. On accept,
`DEC-0042.md` gets `superseded_by: DEC-0099` written into its
frontmatter. The file stays in place — it's still readable, still in
the ledger, just marked as no longer the active link.

### What the chain looks like

```
DEC-0017 (status: superseded)  ──supersedes──→  null
   │
   superseded_by: DEC-0042
   │
DEC-0042 (status: superseded)  ──supersedes──→  DEC-0017
   │
   superseded_by: DEC-0099
   │
DEC-0099 (status: accepted)    ──supersedes──→  DEC-0042
```

Querying:

```bash
cairn mcp call cairn_supersedes_chain '{"decision_id":"DEC-0017"}'
```

Returns the full chain forward to the current binding decision:

```json
[
  { "id": "DEC-0017", "status": "superseded", "supersedes": null },
  { "id": "DEC-0042", "status": "superseded", "supersedes": "DEC-0017" },
  { "id": "DEC-0099", "status": "accepted",   "supersedes": "DEC-0042" }
]
```

### Why this matters

Six months later, when someone asks *"why is the token lifetime so
short? users are complaining"*, the chain is the answer:

- DEC-0017 — original 7-day refresh, default for the framework.
- DEC-0042 — narrowed to 24h for PCI compliance.
- DEC-0099 — narrowed again to 8h after Level 1 escalation.

You can't relax the 8h limit without an explicit new decision —
which forces the conversation to acknowledge what's being walked
back.

The agent reads only the active link (`DEC-0099`) when planning new
work, so context isn't bloated. The chain is queryable on demand
when the historical reasoning is what you need.

### Multi-step revision

Sometimes the chain branches. If `DEC-0042` covered both auth
tokens *and* refresh tokens, and you want to split them so refresh
tokens get a different policy, you write two superseders:

- `DEC-0099` supersedes `DEC-0042` for the auth-token portion (8h).
- `DEC-0100` supersedes `DEC-0042` for the refresh-token portion
  (24h).

`DEC-0042` ends up with `superseded_by: [DEC-0099, DEC-0100]` (yes,
the field can be a list when the split occurs). The chain query
walks both branches.

## Querying decisions

Three primary read tools, picked by what you have.

### You have a list of files → `cairn_in_scope`

The most common query. Used by the agent's tightening step and by
the SessionStart context builder. Returns both DECs and §INVs by
default; pass `types: ["decision"]` or `["invariant"]` to filter.

```
cairn_in_scope({
  path_globs: ["src/auth/jwt.ts", "src/auth/refresh.ts"],
  types: ["decision"],       // optional; default is both
  status: ["accepted"]        // optional; default ["accepted"] for DECs, ["active"] for §INVs
})
```

Returns summaries (no body) of decisions whose `scope_globs`
overlap any of the supplied paths:

```json
[
  {
    "id": "DEC-0042",
    "title": "Auth tokens expire after 24 hours",
    "status": "accepted",
    "scope_globs": ["src/auth/**", "packages/api/src/middleware/auth/**"]
  }
]
```

Pass `status: ["accepted", "superseded"]` to include the historical
chain too.

### You have a specific id → `cairn_decision_get`

Returns the full ADR with frontmatter, assertions, body, and
related invariants:

```
cairn_decision_get({ id: "DEC-0042" })
```

The tool resolves both accepted DECs (canonical zone) and drafts
(`_inbox/`), so you can use the same call for either.

### You have a fuzzy idea → `cairn_search`

Full-text search across all ground state:

```
cairn_search({
  query: "idempotency",
  kinds: ["decision", "invariant"],
  limit: 10
})
```

Returns compact records (~50 tokens each):

```json
[
  { "id": "INV-0091", "kind": "invariant", "title": "Refund operations idempotent", "score": 0.94 },
  { "id": "DEC-0067", "kind": "decision",  "title": "Per-user rate limits use Redis token-bucket", "score": 0.71 }
]
```

Use `cairn_get_full({ id, kind })` to fetch the full body of any
candidate.

### When to use which

- **Tightening a prompt** → `cairn_in_scope` (the agent does this
  automatically; you rarely call it directly).
- **Writing a superseder** → `cairn_decision_get` to read the
  current text first, so the new DEC quotes the exact prior
  language it's replacing.
- **Onboarding a new contributor** → `cairn_search` for the
  domain they're starting in.
- **Auditing what's recorded for a subsystem** →
  `cairn_in_scope` with the subsystem's globs.

## Scope design

Scope is the most important field on a DEC. Get it wrong and either
the decision pollutes irrelevant tasks (too broad) or the decision
misses tasks it should constrain (too narrow). A handful of
heuristics keep this clean.

### The two failure modes

**Too narrow.** You write `scope_globs: ["src/auth/jwt.ts"]` for a
decision about JWT signing. A new file `src/auth/refresh.ts` lands
that handles refresh tokens. The decision doesn't apply (per scope),
so the agent writing `refresh.ts` doesn't load `DEC-0042` and may
silently use a different lifetime. You catch it in review — or you
don't.

**Too broad.** You write `scope_globs: ["src/**"]` for the same
decision. Now every task that touches anything in `src/` loads
`DEC-0042`, even if the task is about UI components or
billing or data-export — none of which care about JWT. The agent's
context fills with irrelevant DECs; the signal-to-noise ratio
drops; tightening becomes noisy.

### Heuristics

1. **Match the subsystem, not the file.** For a decision about
   auth, scope is `src/auth/**` (and any other directory that
   participates in auth). Not just one file; not the whole repo.
2. **Use multiple globs when the subsystem spans directories.**
   Auth often lives in two places — the request middleware and the
   token issuer. Both globs in `scope_globs`.
3. **Include the test directory if the decision is testable.** If
   `INV-0091` says refunds must be idempotent, the test files that
   verify this should be in scope (so the agent reads the
   invariant when writing or modifying the tests).
4. **Don't include build tooling or vendored deps.** A decision
   about coding style doesn't apply to `dist/` or `node_modules/`.
5. **Default to `**` not `*`.** A scope of `src/auth/*` matches
   only direct children, missing `src/auth/strategies/jwt.ts`. Use
   `src/auth/**` unless you specifically want to exclude
   subdirectories.

### Good scopes

```yaml
# Auth subsystem
scope_globs:
  - src/auth/**
  - packages/api/src/middleware/auth/**
  - tests/auth/**

# A decision about how DTOs are validated
scope_globs:
  - packages/api/src/dto/**
  - packages/api/src/routes/**
  - packages/api/test/dto/**

# A naming convention for events
scope_globs:
  - packages/events/**
  - "**/events/*.ts"
```

### Bad scopes

```yaml
# Too narrow — missing related files
scope_globs:
  - src/auth/jwt.ts

# Too broad — pollutes every task
scope_globs:
  - "**"

# Wrong syntax — won't match anything
scope_globs:
  - src/auth/

# Includes generated artifacts
scope_globs:
  - "**"
  - dist/**
  - node_modules/**
```

### Scope evolution

If you discover a scope is wrong after the fact, you don't edit the
existing DEC's `scope_globs` in place — you write a new DEC that
supersedes the old one with the corrected scope. The change is
recorded; future audits see why scope was widened or narrowed.

The exception: a clearly mechanical correction (typo in a glob, a
directory was renamed and the scope wasn't updated). For these,
operator-level edit is fine — the rationale didn't change.

## Patterns and anti-patterns

### Pattern: record the choice, not the implementation

Good DEC titles describe the *choice*:

- "Auth tokens expire after 24 hours."
- "Refunds use Stripe's idempotency-key feature, not a custom
  retry-table."
- "All API responses include `x-request-id`."

Bad DEC titles describe the *code*:

- "Update jwt.ts to set expiresIn to 24h." (transient — the file
  changes, the choice doesn't)
- "Use the new RefundService class." (an implementation, not a
  choice)
- "Add request-id middleware." (likewise)

The DEC outlives any specific implementation. Phrase it so it stays
true even when the code refactors.

### Pattern: rationale answers "why this and not the alternative?"

The body should explicitly mention the alternative considered and
why it was rejected. Future contributors (and future agents) will
reach for the alternative if the DEC doesn't pre-empt it.

```markdown
## Decision
Refund retries use exponential backoff with jitter, max 5 attempts.

## Why not infinite retry?
A retry storm against an unresponsive Stripe instance has caused
bill spikes before (run-RUN-0042). Capping at 5 attempts limits
worst-case spend to bounded N×base.
```

### Anti-pattern: vague scope

`scope_globs: ["src/**"]` is almost never the right answer. If a
rule really applies to all source code (a coding-style rule, an
import convention), it's an invariant, not a decision. Decisions
are subsystem-scoped by nature.

### Anti-pattern: long body

A DEC body should fit on one screen. If you're writing >300 words
of rationale, you're probably bundling multiple decisions — split
them. The agent reads the full body when the DEC is in scope; long
bodies waste the context budget.

### Anti-pattern: stacking many DECs about the same subsystem

If you find yourself writing `DEC-0099`, `DEC-0100`, `DEC-0101` all
about auth tokens (one for lifetime, one for refresh strategy, one
for the rotation policy), consider whether they're actually one
decision split into pieces. Sometimes that's right (the pieces are
genuinely independent). Often it's a sign the underlying decision
hasn't been clearly articulated yet — try to write the umbrella
decision first.

## Practical example: a multi-month decision evolution

The full lifecycle of one decision, illustrating how scope, supersedes,
and the chain compose.

### Day 1: initial choice

You're building the auth subsystem. You record a baseline:

```yaml
id: DEC-0017
title: Auth tokens expire after 7 days
status: accepted
scope_globs:
  - src/auth/**
decided_at: 2026-02-01
```

The body explains: industry-standard refresh window for the
framework defaults; balances UX and security.

### Day 90: PCI audit forces narrowing

Q1 PCI audit lands. Tokens > 24h are out of policy for cardholder
data. You write a superseder:

```yaml
id: DEC-0042
title: Auth tokens expire after 24 hours
status: accepted
scope_globs:
  - src/auth/**
  - packages/api/src/middleware/auth/**     # added — middleware was a separate dir
supersedes: DEC-0017
decided_at: 2026-04-12
```

The body explains the audit findings, references the silent-refresh
strategy. `DEC-0017` is now `superseded_by: DEC-0042`.

You also add machine-checkable assertions for the lifetime:

```yaml
assertions:
  - id: a1
    kind: text_must_match
    pattern: "expiresIn:\\s*'24h'"
    in_globs: ["src/auth/jwt.ts"]
```

So a future commit that silently bumps the lifetime back to 7d
will fail the pre-commit sensor.

### Day 200: Level 1 escalation

You hit 6M transactions, get reclassified to PCI Level 1. The cap
drops to 8h:

```yaml
id: DEC-0099
title: Auth tokens expire after 8 hours (PCI Level 1)
status: accepted
scope_globs:
  - src/auth/**
  - packages/api/src/middleware/auth/**
supersedes: DEC-0042
decided_at: 2026-08-15
```

The body explains Level 1 mandate, links to the silent-refresh
escalation work, notes mobile-client coordination required.

`DEC-0042` is now `superseded_by: DEC-0099`. Assertions are updated
to match `'8h'`.

### Day 220: someone asks "why is the token lifetime so short?"

A new contributor opens an issue. They want to know the history.

You point them at:

```bash
cairn mcp call cairn_supersedes_chain '{"decision_id":"DEC-0017"}'
```

Returns the full chain — DEC-0017 → DEC-0042 → DEC-0099 — with
status, dates, and supersedes pointers. They read the bodies in
order and have the complete history in 5 minutes.

### Day 400: the chain prevents silent regression

A new feature needs longer-lived tokens for a specific use case
(server-to-server). An agent prompted with *"add long-lived tokens
for service-to-service"* loads `DEC-0099` and refuses to touch the
existing token issuer. Instead, the spec it tightens to says:

> Add a separate service-account flow with its own lifetime
> (DEC-0019 covers the 90-day rotation). Do not modify the user
> bearer-token lifetime, which is constrained by DEC-0099.

The chain prevents the new feature from silently relaxing the
PCI-mandated cap. The constraint is enforced not by the agent
remembering — by the agent reading the recorded decision.
